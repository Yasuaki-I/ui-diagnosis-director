# UI診断ディレクター デザインシステム

> このドキュメントは UI診断ディレクター GPTs の出力品質を担保するための「正典」。
> Instructions / 03_pptx_builder.py / サンプル全てがこの規定に従う。

**最終更新**：2026年6月28日（日）
**バージョン**：v1.0（文字数バリデーション導入版）

---

## 0. 設計思想

### コア価値
**「迷わず・即座に・誰にでも分かる形で出力する。」**

### 媒体特性
スライドは**コンテナサイズ固定**の媒体である。
紙やWebと異なり、テキスト量に応じて自動的に余白が伸縮することはない。
→ **テキスト量を物理的にコントロールしなければレイアウトは必ず崩壊する。**

### 編集容易性の絶対要件
エンドユーザーが**自社PPTXフォーマットへコピペ転記**することを最優先。
→ メイリオ統一 / 14pt下限 / 装飾は最小限 / 図形と文字は分離レイヤー。

## 1. 文字数規定（最重要・絶対遵守）

### 1.1 基本ポリシー

| 原則 | 内容 |
|---|---|
| **上限値の絶対遵守** | 規定文字数を1文字でも超えたらバリデーションエラー停止 |
| **省略禁止** | 末尾 `…` での切り詰めは禁止 |
| **要約推奨** | 規定文字内で**要点を絞り込む**ことで対応 |
| **品質優先** | 文字数オーバー時はGPTsが再考し、規定内に要約し直す |

### 1.2 哲学

> **「制約は品質を高める。」**

長文 = 親切ではない。要点を絞り込めない文章 = ディレクターの腕が問われる。
規定文字数内で本質を伝える文章こそ、**経営層・実務層の双方が読める良い文章**である。

### 1.6 バリデーション失敗時の挙動

#### コード側（03_pptx_builder.py）の挙動

```python
def validate_length(text, max_len, field_name):
    """文字数超過時はValueErrorで停止する。"""
    if text is None:
        return ""
    text_str = str(text)
    if len(text_str) > max_len:
        raise ValueError(
            f"\n[文字数超過エラー]\n"
            f"  項目: {field_name}\n"
            f"  上限: {max_len}文字\n"
            f"  実際: {len(text_str)}文字（{len(text_str) - max_len}文字オーバー）\n"
            f"  内容: 「{text_str[:30]}...」\n"
            f"→ GPTs側で規定文字数内に要約し直してください。\n"
            f"  ※末尾「…」での省略は禁止です。"
        )
    return text_str
```

#### GPTs側の対応フロー

```
1. ユーザーから診断依頼を受ける
   ↓
2. 診断結果をDIAGNOSIS辞書に格納
   ↓
3. PPTX生成を実行
   ↓
4. 文字数超過エラーが発生したら：
   - エラー内容を確認（どの項目が何文字オーバーか）
   - 該当箇所を規定文字数内で要約し直す
   - 末尾「…」は使わない
   - 再度PPTX生成を実行
   ↓
5. エラーなく生成完了 → ユーザーに納品
```

## 2. タイポグラフィ規定

### 2.1 フォント

| 用途 | フォント | 理由 |
|---|---|---|
| **全テキスト** | **Meiryo（メイリオ）** | コピペ調和の最重要要件 |
| ❌ 禁止フォント | Hiragino Kaku Gothic ProN | Windowsで文字化け |
| ❌ 禁止フォント | Yu Gothic, Noto Sans JP 等 | 自社PPTXへのコピペで意図と異なるレンダリング |

### 2.2 タイポスケール

| サイズ | 用途 | 行間 | letter-spacing |
|---:|---|---:|---|
| **32pt** | スコア値・ランクなど最重要数値 | 1.0 | -0.5 |
| **24pt** | スライドタイトル・大見出し | 1.1 | 0 |
| **18pt** | セクション見出し・小見出し | 1.2 | 0 |
| **16pt** | 本文（主） | 1.4 | 0 |
| **14pt** | 本文（密度高）・注記 | 1.4 | 0 |

### 2.3 太さ・色のメリハリ

| 階層 | 太さ | 色 |
|---|---|---|
| 最重要数値（スコア・ランク） | **Bold** | `NAVY` または `RED` |
| 見出し | **Bold** | `NAVY` |
| 強調本文 | **Bold** | `TEXT`(#404040) |
| 通常本文 | Regular | `TEXT`(#404040) |
| 補足・注記 | Regular | `SUB_TEXT`(#606060) |

## 3. 配色規定

### 3.1 基本パレット（3色アクセント + 中立色）

| ロール | 色名 | HEX | 用途 |
|---|---|---|---|
| **Primary** | NAVY | `#1C366C` | 見出し・タイトル帯・主要強調 |
| **Alert** | RED | `#D0021B` | 最優先課題・低スコア（1-2点）強調 |
| **Accent** | ORANGE | `#F97316` | 改善方向・POINT帯・行動喚起 |
| Neutral | WHITE | `#FFFFFF` | 背景・カード地色 |
| Neutral | LIGHT_GRAY | `#F4F5F8` | セカンダリ背景・ストライプ |
| Neutral | BORDER_GRAY | `#D0D4DC` | 罫線・区切り |
| Text | TEXT | `#404040` | 本文 |
| Text | SUB_TEXT | `#606060` | 補足・注記 |

### 3.2 透明度バリエーション（次フェーズ：ビジュアル強化で使用）

| バリアント | 計算 | 用途 |
|---|---|---|
| NAVY_LIGHT | NAVY 18% 透過上塗り（白下地） | カード薄塗り背景 |
| NAVY_PALE | NAVY 5% 透過上塗り（白下地） | スライド全体の極薄背景 |
| ORANGE_LIGHT | ORANGE 12% 透過上塗り（白下地） | POINT帯背景 |
| RED_LIGHT | RED 10% 透過上塗り（白下地） | 警告ハイライト背景 |

### 3.3 3色制限ルール

**アクセントカラーは1スライドにつき最大2色まで**（NAVY + 1色）。
3色すべて使うとビジュアルが散る。原則：
- 通常コンテンツ：NAVY基調
- 課題ハイライト：NAVY + RED
- 改善・行動喚起：NAVY + ORANGE

## 5. 項目数規定

### 5.1 固定数の規定

| 項目 | 件数 | 例外 |
|---|---:|---|
| 評価軸（スコア） | **10項目固定** | バナー診断時はN/Aで5項目 |
| 強み（C-1） | **3項目固定** | なし |
| 最優先課題（C-1） | **3項目固定** | なし |
| 改善提案（C-2） | **5項目固定** | バナー診断時は3項目可 |
| Before/Afterハイライト（C-3） | **3項目固定** | なし |
| 行動フローステップ（C-3） | **3〜5ステップ** | LP構造による |
| LP構造マップセクション（C-3） | **5〜7セクション** | LP構造による |

### 5.2 項目数の超過・不足時の挙動

- **超過時**：バリデーションエラーで停止 → GPTsが上位N件に絞り込み再実行
- **不足時**：バリデーションエラーで停止 → GPTsが追加項目を生成し再実行
- **N/A時**：score = `'－'`、comment = `'バナー診断のため対象外'` で固定
